# JD 评估规则

## 用途

AI 助手读到一个岗位的 JD 后，按此规则给岗位打分分档，并输出结构化的评估结果，
供下一步定制简历和写招呼语使用。

## 输入

AI 助手会拿到两样东西：

1. **JD 文本**：OCR 识别的岗位描述（可能有换行/空格噪音，但足够读懂），
   或者用户直接粘贴的 JD 原文。
2. **候选人的母版简历内容**：存在 `cv/example_content.yaml` 里（或用户自己
   写的 `cv/content_*.yaml`），包括基本信息、教育、项目、实习、技能。

## 评估步骤

### 第一步：拆 JD

把 JD 拆成两部分：

- **硬性要求**：学历、实习时长、到岗天数、必需技能、领域经验。这些是门槛，
  不满足就不建议投。
- **加分项**：有更好、没有也不影响过筛的。比如「有 XX 经验者优先」。

### 第二步：逐条比对母版简历

拿着硬性要求和加分项，跟候选人的母版简历逐条对比。

**诚实原则（最重要）**：
- 可量化的要求必须如实核对。JD 要求 6 个月实习，候选人只写 3 个月，那就
  是缺口，如实标。
- 不编造不存在的经历去凑 JD 的要求。
- 「方向接近但没直接做过」的，标为「可用但需补强」，不要硬说成「匹配」。
- 「不确定 JD 这条要求指什么」的，标为不确定，不强行解读。

### 第三步：打分

按以下维度打 0-100 分：

| 维度 | 权重 | 说明 |
|------|------|------|
| 内容匹配度 | 60% | 项目/实习经历是否覆盖 JD 的核心职责领域。覆盖越多越高。 |
| 技能匹配度 | 20% | JD 明确要求的工具/技能在简历中出现了多少。 |
| 硬门槛 | 10% | 学历、实习时长、到岗天数等硬性条件是否满足。有一条不满足扣 10 分，扣到 0 为止。 |
| 加分项匹配 | 10% | 加分项命中几条。 |

### 第四步：分档

| 分数 | 分档 | 含义 |
|------|------|------|
| ≥70 | 重点定制 | 高度匹配，值得深度定制简历、写针对性招呼语 |
| 40-69 | 普通模板 | 有一定匹配度，用通用模板，不深度定制 |
| <40 | 不建议投 | 硬缺口太多或方向不匹配 |

### 第五步：诚实标注缺口

分档后，无论哪一档都要诚实列出硬缺口。

格式：「[要求] → [实际] → [缺口说明]」

```
例：要求「每周5天」→ 简历写「4天/周」→ 差1天，如可协调可忽略
例：要求「计算机专业优先」→ 本科学国际贸易 → 非对口，但有AI微专业方向接近
```

## 输出格式

对每个岗位输出一个结构块，格式如下（无论给用户看还是给下一个环节用）：

```
## [公司] — [岗位]

匹配分：xx/100 | 分档：重点定制 / 普通模板 / 不建议投

### 命中点
- 简历的哪段经历 → 对上 JD 的哪条要求（具体到行文）
- ...

### 硬门槛缺口
- [要求] → [实际] → [缺口说明]
- ...

### 简历强调建议
- 个人定位方向建议写什么（扣 JD 核心关键词）
- 建议保留哪些项目/实习（按与 JD 匹配度排序）
- 建议弱化或删掉哪些内容（方向不一致或较弱）
- 建议突出的具体数字或成果

### 推荐钩子作品
从候选人公开作品列表里选一个最对口的。
说明选它的理由（这个作品的核心能力 / 交付物正好扣 JD 的哪条要求）。
```

## 注意事项

- 这个评估是给 AI 助手自己读的规则，不是给终端用户的文档。评估结果由助手
  在对话中输出给用户。
- 如果用户提供了多份 JD，逐个评估后按匹配分高低排序输出。
- 注意「分档」和「是否投递」是分开的事。即使分档是「普通模板」，用户
  也可能选择投递；即使分档是「重点定制」，也可能因硬缺口太大被建议放弃。
- 分档为「不建议投」的岗位，助手应明确给出理由，但最终投不投由用户决定。
